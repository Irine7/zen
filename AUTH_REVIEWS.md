# Рекомендации по улучшению системы авторизации (Zen Garden)

В данном документе собраны рекомендации по доработке системы аутентификации и авторизации для перехода от прототипа к production-ready решению.

## 1. Безопасность (Security)

### Хранение токенов
*   **Проблема:** Использование `localStorage` делает приложение уязвимым к XSS-атакaм.
*   **Решение:** Переход на **HttpOnly, Secure Cookies**. Это скроет токен от JavaScript и обеспечит его автоматическую передачу браузером.

### Хеширование и валидация
*   **Валидация данных:** Добавить строгие правила на бэкенде (`AuthResolver`) и фронтенде:
    *   Проверка сложности пароля (минимум 8 символов, наличие цифр и спецсимволов).
    *   Валидация формата Email (regex).
*   **Защита от перебора (Brute-force):** Внедрить **Rate Limiting** (ограничение количества запросов) для эндпоинтов `signIn` и `signUp`.

## 2. Улучшение UX и жизненный цикл сессии

### Механизм Refresh Tokens
*   **Проблема:** Текущий JWT живет 7 дней, после чего сессия обрывается.
*   **Решение:** Внедрить пару токенов:
    1.  **Access Token:** Короткоживущий (15–60 мин) для запросов к API.
    2.  **Refresh Token:** Долгоживущий, хранящийся в БД и в HttpOnly Cookie, для получения нового Access токена без повторного ввода пароля.

### Управление аккаунтом
*   **Logout (Выход):** Реализовать функцию выхода, которая будет очищать токены на клиенте и инвалидировать их на сервере.
*   **Восстановление пароля:** Добавить флоу "Забыл пароль" с отправкой временного токена на Email.
*   **Подтверждение Email:** Реализовать подтверждение почты при регистрации для верификации пользователей.

## 3. Архитектурные улучшения

### Надежность AuthGuard
*   **Проблема:** Текущий `AuthGuard` проверяет только наличие строки в хранилище.
*   **Решение:**
    *   Добавить проверку срока действия токена (exp claim) на фронтенде.
    *   При инициализации приложения выполнять запрос `me` к API для проверки валидности сессии на сервере.

### Обработка ошибок (Global Auth Interceptor)
*   Добавить в Apollo Client `onError` линк, который при получении ошибки `UNAUTHENTICATED` (401) будет автоматически:
    1.  Пытаться обновить токен (если есть Refresh Token).
    2.  Либо разлогинивать пользователя и перенаправлять на `/login`.

---

## 4. Будущее развитие

*   **OAuth / Социальный вход:** Интеграция с Google, GitHub или Apple.
*   **Ролевая модель (RBAC):** Если появятся разные типы пользователей (например, модераторы), стоит внедрить систему ролей и прав доступа.
*   **Двухфакторная аутентификация (2FA):** Для повышения безопасности аккаунтов.
